# Plan: Rediseño integral de nm_suite (Hub + Suite + Instaladores)

## Context

El repositorio `nm_suite` es una aplicación PyQt6 de salud mental compuesta por dos productos (Suite paciente y Hub profesional) más sus instaladores/desinstaladores. El sistema de diseño v3 actual (teal/violeta) funciona pero presenta:

- **Dos bugs visuales confirmados en código**: `timer_qt.py:215` duplica el texto "Lista para empezar" (`_state_chip` + `_canvas.set_data` pintan el mismo string), y `rutina_qt.py:469-489` carga tareas sin deduplicación lo que produce items repetidos visualmente.
- **Inconsistencias de tokens**: `V3_SPACE["md"]=12` (`theme.py`) vs `SPACE["md"]=16` (`theme_qt.py`). Sistemas paralelos `V3_SHADOWS` vs `SHADOWS`, `V3_GRADIENTS` vs `GRADIENTS`.
- **Componentes faltantes** para un design system maduro: `NMModal`, `NMDialog`, `NMTooltip`, `NMDivider`, `NMAvatar`, `NMErrorState`, `NMSearchInput`, `NMTextArea`, `NMSectionHeader`, `NMTabs`.
- **Stylesheets globales incompletos**: faltan `QMenu`, `QDialog`, `QListWidget`, `QTableWidget`, `QSpinBox`, etc.
- **Tema** dual con acentos distintos por modo (teal en light, violet en dark) que confunde la identidad visual.
- **Tipografía** body a 13px — por debajo del umbral cómodo para una app clínica.
- **Sidebar Suite invisible** en runtime: `_SidePanel` se confunde con el shell background.
- **Estados de accesibilidad** (focus rings, accessible names, touch targets) parcialmente implementados.

**Decisiones del usuario (confirmadas)**:
1. Renovación cromática total — salir del eje teal/violet.
2. Crear wrappers `NM*` dedicados para los componentes faltantes (no solo stylesheets sobre nativos).
3. Rediseño visual nuevo, no preservación del sistema actual (refactor de tokens, no solo cambio de hex).
4. Bug fixes en Fase 0 antes de cualquier cambio visual.

**Outcome esperado**: producto unificado visualmente (Hub + Suite + installers), responsive real, accesible (WCAG AA), con sistema de diseño centralizado y un único acento cromático coherente. Lógica de negocio, datos, sincronización Supabase y proceso de build **intactos**.

## Riesgos técnicos y mitigaciones

| Riesgo | Severidad | Mitigación |
|---|---|---|
| Cambiar `V3_LIGHT`/`V3_DARK` rompe los 16 consumidores legacy vía bridge | 🔴 ALTO | Mantener las MISMAS CLAVES en V3_LIGHT/V3_DARK (solo cambiar valores hex). Actualizar `_bridge_dark()`/`_bridge_light()` en el mismo commit. Verificar que `COLORS["dark_hybrid"]` y `COLORS["light_hybrid"]` siguen exponiendo todas las claves legacy esperadas. |
| `ThemeManager` singleton + 52 componentes con conexión manual a `theme_changed` | 🟠 MEDIO | Cualquier componente nuevo DEBE conectarse a `ThemeManager.instance().theme_changed.connect(self._apply_theme)` en `__init__`. Patrón establecido. |
| Qt6 no acepta `rgba()` en todos los QSS properties | 🟠 MEDIO | Usar `parse_rgba()` + `qcolor_to_rgba_css()` (ya existen en `theme_qt.py:1398, 1262`). No usar rgba() literal en stylesheets. |
| `paintEvent` custom en 70+ componentes ignora cambios de QSS | 🟠 MEDIO | Para componentes con paint custom (NMCard, NMButton, NMModuleRing, etc.), los tokens nuevos se leen vía `v3c()` / `C()` en cada paint. Verificar manualmente que cada paint custom lee del tema actualizado. |
| Bug Rutina puede ser causado por `_load_tasks` llamado dos veces y no por duplicación de DB | 🟠 MEDIO | Antes de fixear, instrumentar con log para confirmar si es doble call o duplicación en data source. Solución correcta depende de la causa. |
| Fix Timer eliminando `_state_chip` puede romper algún caller externo | 🟡 BAJO | Buscar referencias a `_state_chip` antes de eliminar; si hay, usar setter alternativo o redirigir a `_canvas.set_data`. |
| `installer.py` siempre dark mode con `stylesheet_installer()` | 🟡 BAJO | No aplicar el theme toggle en installers. Mantener `stylesheet_installer()` separado pero usando los nuevos tokens del modo dark. |
| 6443 líneas en `components_qt.py` — errores de sintaxis colapsan toda la app | 🟡 BAJO | Cada cambio en components_qt.py va seguido de `python -m py_compile shared/components_qt.py` antes de continuar. |
| `responsive_columns` existe en components_qt.py (no en theme_qt.py como reportó el agente) | 🟢 BAJO | Verificar ubicación antes de mover. Si está en components_qt.py, dejarla ahí o mover a theme_qt.py como parte de Fase 7. |
| Build PyInstaller puede romperse si imports nuevos no están en `requirements.txt` | 🟢 BAJO | No agregar dependencias nuevas. Todo se hace con PyQt6 + lo que ya está. |
| Capturas viejas en `REDESIGN/03_current_app/` quedan desactualizadas | 🟢 BAJO | Se acepta y se documenta. Al final de Fase 8, regenerar capturas si el usuario lo pide. |

## Proceso obligatorio entre fases

Después de **cada** fase:
1. **Mostrar archivos modificados** (`git status` + `git diff --stat`).
2. **Resumen de cambios** en 3-5 líneas.
3. **Validación de sintaxis**: `python -m py_compile <archivos>`.
4. **Smoke test**: correr la app afectada (`python app/main_qt.py` o `python hub/main_qt.py`) y verificar que arranca sin excepción.
5. **Capturas frescas** de las pantallas afectadas (no comparar con `REDESIGN/03_current_app/` viejo).
6. **Diff resumido** de los cambios significativos.
7. **Pausa explícita** para que el usuario revise antes de continuar a la siguiente fase.
